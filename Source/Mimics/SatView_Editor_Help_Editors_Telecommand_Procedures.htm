The design of Telecommand Procedures involves two parts as there are the definition of the execution paths or program flow (which is usually done in an interactive graphical way) and the specification of additional properties necessary to generate the executable code. These two parts are reflected by the two panes of the interface: The left one hosting the graphical area and the right one properties area.

  3. Push the OK button.

The configurable properties of a telecommand procedure apply to the panes where the thread execution flow can be defined in an interactive and graphical way.

  1. Open the TC Procedure Properties Dialog Box if it is not open yet:

  2. Design & Edit a Telecommand Procedure

The Telecommand Procedure Display provides a graphical environment for defining operating telecommand procedures. It is aimed at the operations preparation environment and provide a tool to prepare telecommand procedures for subsequent execution in the related environment. Further capabilities like editing and basic debugging are supported too.
The preparation of telecommand procedures involves the following stages:

  • Creating and configuring the telecommand procedure structure
  • Testing the flow paths through the telecommand procedure (debugging)

A telecommand procedure should not be released for use until it has been fully validated. The telecommand procedure display should be viewed as development environment i.e. as the first stage in creating and maintaining a validated operating telecommand procedure. An automated telecommand procedure created in this way fulfills the same role as a traditional ‘manual’ telecommand procedure. It typically sends telecommands, monitors parameters, creates and responds to events. The structure and flow of the telecommand procedure is represented by certain components (called flow-control steps and action-object steps), which all have a graphical representation in the telecommand procedure display. Some steps allow expressions to be used which are evaluated at run-time to provide values which are then used. For example, each telecommand procedure step has a trigger condition expression which is repeatedly evaluated until it returns true, allowing the execution of the step to proceed.
A telecommand procedure display is split into two panes, a left one that contains the flowchart (a graphical representation of the execution path) and a right one that consists of various tabs where additional information can be entered. It is possible to develop a telecommand procedure without the support of the graphical interface.

Opening a Telecommand Procedure

  • Create or open the telecommand procedure to be edited:
    Select the New... item on the application button's menu, choose the TC Procedure item in the Type combo box of the dialog box and design the telecommand procedure as required.
    Select the Open è TC Procedure... item on the application button's menu and choose the procedure to be opened in the TC Procedure Open Dialog Box.

Saving a Telecommand Procedure

  • Select the Save item on the application button's menu to save an existing one.
    Use the Save As... item to save a new one or an existing one with a different name.

Specifying the operational Properties

The telecommand procedure is the structure that holds all the components that comprise a single operating procedure. It can be viewed as a container where all the elements of the procedure are stored. In addition to this main function, a telecommand procedure can also be configured in its behavior. An example may be the default way in which a procedure executes (e.g. in automatic mode, or in single step mode where each step must be manually authorized by an operator). The most significant features of the telecommand procedure are its arguments and local variables and, of course, its associated code. Dubugging support is also available.

Specifying the formal Arguments

With the Arguments property page the calling arguments of a telecommand procedure are defined: When a telecommand procedure is executed, it is possible to define a number of values which are passed in to the procedure at run time. The values passed to the telecommand procedure when instantiating it for execution are referred to as arguments. Within the definition of the telecommand procedure, the number and type of arguments which the procedure takes at run time must be declared. A procedure may have zero or more arguments. In order for the telecommand procedure to execute, each argument must be given a value at the start of execution.
An argument can receive a value in a number of ways when execution of the telecommand procedure commences:

  • If the telecommand procedure is called from within another one the value of the arguments is specified by the calling procedure.
  • A telecommand procedure started manually must have argument values provided by the operator configuring the procedure instance.
  • In the case of a scheduled telecommand procedure execution the argument values can consist of predefined values but also of expressions that are evaluated when the execution begins.

Alternatively, each argument can be given a default value in the editor that will be used if a value is not given at run time.
Within a telecommand procedure the value of an argument can be referred to or changed. An expression may refer to the value of an argument or a 'Set-Argument' action-object step can give an argument a new value. Arguments are typically used to make a telecommand procedure more general. For example a spacecraft maneuvering procedure may use its arguments to represent thruster burn times.
Important: For each telecommand procedure its arguments and variables must have unique, mutual exclusive names.

Defining the local Variables

With the Variables property page the local variables of a telecommand procedure are defined: A variable is defined and used within the telecommand procedure to store temporary values. It is only available within the current telecommand procedure. There may be zero or more local variables. A variable can be given a default value that is used as the initial value of the variable when the telecommand procedure starts to execute. Note that if the variable is not given a default value, it will not have a value when the procedure begins to execute. Before a telecommand procedure refers to the value of a variable, its value must be set. A variable can be given a value by a 'Set-Variables' action-object step, or by prompting an operator to provide a value though the use of a 'User-Interaction' action-object step.
Important: Arguments and variables for a particular telecommand procedure must have unique, mutual exclusive names.

Defining & Configuring the Threads

With the Threads property page the various required threads can be defined: A thread is a sequence of execution. To be useful it must contain one or more steps. A thread represents a main flow path through a telecommand procedure and in this context there are two types of them:

  • One Primary thread
  • Zero or more secondary threads

A telecommand procedure must have a primary thread. When a new telecommand procedure is created it automatically contains a primary thread which cannot be deleted. A telecommand procedure may have any number of secondary threads. The key features of a thread are the auto-enable attribute, which determines if the thread is initially active, and the activation expression, which can be used to trigger the initial execution of the thread. Primary and secondary threads are functionally equivalent. They have the same attributes and are constructed and executed in the same way. The difference is in the way they are used within a telecommand procedure.

Primary Thread
The primary thread is, by convention, the main thread of execution of the telecommand procedure. It should normally be auto-enabled and commence executing immediately. There may be circumstances in which an activation expression is required, although this is unusual for a primary thread.
It also contains steps that implement the main function of the telecommand procedure.

Secondary Threads
Secondary threads contain a number of steps. They may be initially disabled, and then enabled at an appropriate point by a 'Thread-Control' action-object step in the primary thread. A secondary thread may also have an activation expression that causes the thread to be activated by a logical expression that detects a failure in the execution of a primary thread.
The use of enabling and disabling a secondary thread from the primary thread, and an activation expression in the secondary thread, enables secondary threads to be used as contingencies by the primary thread. In a critical section of the primary thread the appropriate secondary thread can be enabled to allow the activation expression to detect a failure in the critical section, possibly by monitoring values of relevant parameters. When triggered, the secondary thread can suspend the primary thread, carry out corrective actions and then resume the primary thread.
Whenever a thread terminates its execution in a disabled state it re-enters and waits again for its activation expression (if one is existing) to become true.

Configuring the Flow-Control Steps & Action-Object Steps

A telecommand procedure contains one or more threads, each of them consisting of one or more steps. These steps are divided into two types: Flow-control steps on one hand and action-object steps on the other one. Flow-control steps provide the means of implementing multiple paths through a procedure, usually based on a logical test. This step type performs the role of programming constructs such as 'repeat', 'while', 'for', 'case' as well as others. Flow-control steps also have an associated invariant condition, a logical expression that must remain true throughout the execution of the step or the execution phase will fail. An action-object step carries out some kind of action as the step executes.
For example:

  • Send a telecommand
  • Generate an interaction with the operator
  • Set the value of a telecommand parameter

Each of the steps is divided into three different phases: Trigger, body and confirmation. Every step has the same configuration available for its trigger and confirmation phases but those phases are optional. The body phase is the prime function of the step and has a different configuration depending on the step. Inside the flowchart the tree step phases are symbolized by a small triangle pointing into the direction of the execution. They change their color depending on the (successful) execution of the associated phase: Green for success, red for a failure, yellow for the current execution or black for a forced result.
The following steps are supported:

Type

Step

Function

Flow-control

Branch

Provides a branch facility commonly referred to as a ‘IF-THEN-ELSE’ statement. The 'true' execution branch (sub-thread) will be executed if the branch expression evaluates to true; otherwise the 'false' sub-thread will be executed. Providing no expression always results in the execution of the 'true' sub-thread.

 

Switch

This step corresponds with a 'CASE' statement. An expression can be supplied for each branch in order to determine which one should be executed. If more than one expression results to true, the first one i.e. its associated branch is selected for execution. Providing no expression for a particular branch always results in its execution.

 

For-Loop

Implements a loop block executing a specified number of times. It corresponds with a 'FOR' statement.

 

While-Loop

Implements a loop block executing as many times as a supplied expression results to true. It corresponds with a 'WHILE' statement.

 

Do-Loop

Implements a loop block executing as many times as a supplied expression results to true. In contrast to the previous loop the expression is tested after executing its associated block. It corresponds with a 'DO' statement.

 

Block

Allows one or more steps to be grouped into a block and to be collapsed/expanded inside the flowchart.

 

Goto

Redirects the execution path to a specified label. It corresponds with a 'GOTO' statement.

This step type has no execution phases.

Configuring the debugging Options

Debugging is an important step in the process of developing a telecommand procedure.
The following features are supported to help testing:

BreakpointsSuspend the execution when a breakpoint is reached.
Breakpoints can be set at each (named) step or source code and can have a conditional nature.
Test ProfilesAllow the definition of telemetry parameter groups attached with a predefined value. This is useful to simulate various spacecraft configurations and can also be used in the context of the telecommand pre-execution verification and execution verification checks. Test profiles can be injected at each (named) step.
IdentifiersWhenever the execution is suspended either by a breakpoint or manually all arguments, variables, telemetry parameters and telecommand parameters used inside the telecommand procedure are listed and their values can be modified when necessary.

Setting a Breakpoint

  1. Open the Debug property page.
  2. Select the thread in the Thread combo box that should get a breakpoint.
  3. Inside the Properties group box specify the location for the breakpoint by choosing it in the Before step combo box.
    Note:
    Only steps that have a name can be attached with a breakpoint. Read how to configure the various flow-control steps and action-object steps in order to learn how to name a step.
  4. Specify under what circumstances a breakpoint should suspend the execution:
    AlwaysBreaks always
    At failed triggerBreaks only when the trigger phase failed
    At failed bodyBreaks only when the body phase failed
    At failed confirmationBreaks only when the confirmation phase failed
  5. Mark the Disabled check box if a breakpoint is temporarily not used but should not be deleted.
  6. Push the Add button.

Alternative way to set a Breakpoint

  1. Select the pane (on the left part of the telecommand procedure display) referring to the thread in those execution flow a breakpoint should be set.
  2. Activate the step to which a breakpoint should be attached to by clicking onto it. It will get highlighted when it is activated.
  3. Click with the mouse button onto the left gray border inside the flowchart pane at the vertical position of the activated step.
Note:

Breakpoints can also be set on lines of source code by clicking at the desired location inside the debug margin of the source code pane.

  1. Open the Debug property page.
  2. Select the thread in the Thread combo box that should get a breakpoint changed.
  3. Select the breakpoint that should be enabled or disabled in the Breakpoints list view.
  4. Mark the Disabled check box to disable or uncheck it to re-enable the breakpoint.
    Note:
    Other properties of the breakpoint can also be changed this way.

  1. Select the pane (on the left part of the telecommand procedure display) referring to the thread in those execution flow a breakpoint should be changed.

  3. Check the Disable menu item in the appearing floating menu to disable or uncheck it to re-enable the breakpoint.
    Note:
    Other properties of the breakpoint can also be changed this way.

Removing a Breakpoint

  1. Open the Debug property page.
  2. Select the thread in the Thread combo box that should get a breakpoint deleted.
  3. Select the breakpoint that should be deleted in the Breakpoints list view.
  4. Push the Remove button to delete it. Use the Remove All button to delete all breakpoints.

  1. Select the pane (on the left part of the telecommand procedure display) referring to the thread in those execution flow a breakpoint should be deleted.
  2. Click onto the breakpoint to be deleted inside the left margin of the flowchart pane.

Monitoring & Controlling the Telecommand Stack

The final purpose of telecommand procedures is, of course, the generation of telecommands. All telecommands that are scheduled and released by the procedure are listed in a stack together with some status information. This feature also supports the pre-execution verification checks and the execution verification checks. By logging the output of the release queue to a file, extended control about the correct functioning of the telecommand procedure can be obtained.

Browsing the Telecommand Stack

  1. Open the TC Stack property page.
  2. All scheduled and released telecommands are listed in the Telecommands list view with the following formatting:

    Column

    Description

    No.

    Displays the entry number of a telecommand on the stack.

    Name

    Identifies the telecommand.
    If it is part of a telecommand sequence the name of the sequence is appended in brackets.

    Description

    Describes the telecommand in more detail.

    Arguments

    Shows all non-fixed arguments of the telecommand together with their current value.
    Note:
    Not all arguments require a value. Some argument values are created automatically.

    Flags

    Indicates if a telecommand is grouped with the previous entry ('G') or if it is part of a block ('Bn' where n is the entry number in the block).

    TC Packet

    Shows the telecommand packet with which the current telecommand is encoded.

    Release Time

    Shows the time when the telecommand was released.

    Rel. Offset

    Indicates the release time offset with respect to the start of the telecommand sequence. It is used to calculate the absolute release time.

    Execution Time

    Shows the absolute on-board execution time of the telecommand.

    Exec. Offset

    Indicates the on-board execution time offset with respect to the start of the telecommand sequence. It is used to calculate the absolute execution time.

    Authorization

    Indicates the type of authorization required.

    Status

    Shows the current release status and verification status of the telecommand.

    Result

    Shows the result of the various checks or if the release of the telecommand was aborted.

Suspending or Resuming the Telecommand Release Queue

  1. Open the TC Stack property page.
  2. Push the Suspend button at the desired time i.e. when there are telecommands in the release queue.
    Note:
    Suspending the telecommand stack will prevent any telecommands from being released. Execution verification checks however will still continue to be done.
  3. Push the Resume button in order to continue the scheduling of the telecommand stack.
Note:

Suspending the telecommand stack for a too long time may result in expired telecommands.

Manipulating the Contents of the Telecommand Release Queue

  1. Open the TC Stack property page.
  2. Select the scheduled telecommand to be removed from the release queue and push the Remove button. Use the Remove All button to delete all scheduled telecommands.
Note:

Released telecommands obviously cannot be removed from the stack because they not part of the release queue any more.

Enabling or Disabling the Release Wait Mode

  1. Open the TC Stack property page.
  2. Mark the Release Wait Mode check box in order to enable telecommands to wait for their release in case the telecommand pre-execution condition is not satisfied. The number of seconds to wait can be specified in the associated spin box.
Note:

This option does not apply to the on-board execution time of telecommands.

Enabling or Disabling the Pre-execution Verification Checks

  1. Open the TC Stack property page.
  2. Mark the Disable pre-execution verification checks check box in order to disable any pre-execution verification checks.

Enabling or Disabling the Execution Verification Checks

  1. Open the TC Stack property page.
  2. Mark the Disable execution verification checks check box in order to disable any execution verification checks.

Logging all Output to a File

  1. Open the TC Stack property page.
  2. Mark the Log to file check box and enter a file name into the associated edit field. All released telecommands will be written to the specified file the next time the telecommand procedure is tested.
Note:

Make sure to save any modifications by selecting the Save item on the application button's menu. Performing a compilation automatically results in saving all changes.

Coding Guidelines

The source code of a telecommand procedure always consists of a hosting procedure containing one or more thread blocks.

The syntax looks as follows:

TELECOMMAND PROCEDURE 'Name'  [(argument-declaration-list)]
[variable-declarations]

BEGIN

{{THREAD 'Thread-Name'

[TCPARAMETERS Ti...Tj]

[PARAMETERS Pi...Pj]

[VOLATILE Pk...Pl]

BEGIN

TPL Code

END}}

END

Notation:

 

Name

The name of the telecommand procedure.

arg-decl-list

Argument declaration (C++ syntax).

var-decls

Local variables declaration (C++ syntax).

Ti

The tag of a telecommand parameter.

Pi

The tag of a telemetry parameter.

[x]

x occurs zero times or once.

{x}

x occurs zero or more times.

{{x}}

x occurs once or more times.

By using the VOLATILE keyword the subsequent enumerated telemetry parameters are not checked whether or not they are initialized (i.e. do have a value). Each of the specified parameters must be contained in the list declared with the PARAMETERS keyword and may have a random value. By default, the procedure is not executed whenever one of the parameters Pi...Pj has no value.
With the VOLATILE keyword specified, the calculation in the procedure is performed although one or more of the parameters Pk...Pl have no (or a random) value.

Compile & Test a Telecommand Procedure

Telecommand procedures should be tested and debugged before they are used in an operational environment. This is achieved by predefining telemetry conditions to be tested and then executing the associated source code.

  1. Create or open the telecommand procedure to be compiled and debugged:
    Select the New... item on the application button's menu, choose the TC Procedure item in the Type combo box of the dialog box and design the procedure.
    Select the Open è TC Procedure... item on the application button's menu and choose the procedure to be opened in the TC Procedure Open Dialog Box.
  2. Push the Compile button inside the Compilation panel of the Build ribbon bar category.
    Note:
    All output generated by the compiler is displayed in the Build page of the Output tool window.
  3. Correct any reported errors or warnings by double-clicking onto a message in the Build page of the Output tool window. This highlights the line in the source code where the problem is located.
    If required, some help can eventually be obtained by reading more about:
    • Telecommand Procedure Language (TPL)
    • Compiler Basics

Debugging a Telecommand Procedure

A telecommand procedure needs to be compiled first before any debugging can take place. If the executable code to be debugged is not up-to-date, a dialog box will appear asking for a compilation to be done first.     Action

    Steps into the next execution control block. This is possible in suspended mode only.

Import Telecommand Procedures

Telecommand procedures can be shared between distributed databases through importing and exporting.

  1. Open the TC Procedures Import Dialog Box if it is not open yet:
    Select the Import è TC Procedure... item on the application button's menu.
  2. Enter the name of the file that contains the telecommand procedures to be imported into the Filename edit field.
    Use the Browse... button in order to search for the file.
  3. Select the items to be imported inside the list box and specify how to import them by either checking the Default or Custom radio button. By using the Custom radio button it is possible to import a single procedure with another name and description then when it was exported.
    Note:
    Importing a telecommand procedure with the same name than an existing one will overwrite it.
  4. Push the OK button.
Note:

The same result can also be achieved by right-clicking onto the TC Procedures component inside the Procedures page of the Workspace tool window and selecting the Import... item of the floating menu.

Export Telecommand Procedures

Telecommand procedures can be shared between distributed databases through importing and exporting.

  1. Open the TC Procedures Export Dialog Box if it is not open yet:
    Select the Export è TC Procedure... item on the application button's menu.
  2. Choose the telecommand procedures to be exported in the Directory list box.
  3. Specify how to export the telecommand procedures by either checking the Default, Custom or Source Code radio button. By using the Custom radio button it is possible to export a procedure with a different name and description. The Source Code option writes the source code only to the specified file.
    Note:
    The Custom and Source Code options are available only when a single telecommand procedure is selected for export.
  4. Enter the name of the file to contain the telecommand procedures to be exported into the Filename edit field.
    Use the Browse... button in order to find an existing file.
  5. Push the OK button.
Note:

The same result can also be achieved by right-clicking onto the TC Procedures component inside the Procedures page of the Workspace tool window and selecting the Export... item of the floating menu.
